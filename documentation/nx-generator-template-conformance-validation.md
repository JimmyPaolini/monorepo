# NX Generator Template Conformance Validation

A pattern for validating that files generated by NX generators remain structurally intact after developers add their own implementation code.

---

## Problem

NX generators scaffold files from EJS templates. Once generated, developers fill in implementation details — adding imports, properties, methods, etc. There's currently no way to assert that the structural skeleton defined by the template is still intact after those edits.

The goal is a validator that:

- Allows free content to be added anywhere in the file
- Asserts that the template's structural markers are still present and in order
- Can run in tests and/or CI

---

## Solution: Emoji Section Markers

Use emoji comments in EJS templates to define named sections. These comments survive rendering and become the structural anchors that the validator checks for.

### Convention

```typescript
// 🔒 Private
// 🌍 Public
// 🏗️ Constructor
// 🪝 Lifecycle
// 🔧 Methods
```

Use **emoji + descriptive noun phrase** to keep markers unique and self-documenting. Avoid generic single-emoji markers (e.g. `// ✅`) that may collide with other code comments.

### Template Example

```typescript
// files/src/__name__.service.ts__template__

import { Injectable } from '@angular/core';

// 🌍 Imports

@Injectable({ providedIn: 'root' })
export class <%= className %>Service {

  // 🔒 Private

  // 🌍 Public

  // 🏗️ Constructor
  constructor() {
    // 🪝 Init
  }

  // 🔧 Methods
}
```

### Generated File (before implementation)

```typescript
import { Injectable } from '@angular/core';

// 🌍 Imports

@Injectable({ providedIn: 'root' })
export class UserService {

  // 🔒 Private

  // 🌍 Public

  // 🏗️ Constructor
  constructor() {
    // 🪝 Init
  }

  // 🔧 Methods
}
```

### After Developer Implementation

```typescript
import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';

// 🌍 Imports

@Injectable({ providedIn: 'root' })
export class UserService {

  private baseUrl = '/api/users';
  // 🔒 Private

  isReady = false;
  // 🌍 Public

  // 🏗️ Constructor
  constructor(private http: HttpClient) {
    this.isReady = true;
    // 🪝 Init
  }

  getAll() {
    return this.http.get(this.baseUrl);
  }
  // 🔧 Methods
}
```

The validator checks that all emoji markers from the template are still present in the file, in order. Everything else is treated as free content.

## Implementation

### `tools/validators/template-conformance.ts`

```typescript
import * as ejs from 'ejs';
import * as fs from 'fs';

const MARKER_PATTERN = /\/\/\s*\p{Emoji_Presentation}.+/gu;

/**
 * Extracts all emoji-comment markers from a string, in order.
 */
function extractMarkers(content: string): string[] {
  return [...content.matchAll(MARKER_PATTERN)].map((m) => m[0].trim());
}

/**
 * Renders an EJS template with the given variables and returns
 * the ordered list of structural markers it defines.
 */
export function buildExpectedMarkers(
  templatePath: string,
  vars: Record<string, unknown>
): string[] {
  const raw = fs.readFileSync(templatePath, 'utf-8');
  const rendered = ejs.render(raw, vars);
  return extractMarkers(rendered);
}

/**
 * Validates that a file contains all structural markers from the template,
 * in order. Free content is permitted anywhere between markers.
 *
 * @returns An object with `valid` (boolean) and `errors` (string[])
 */
export function validateConformance(
  filePath: string,
  templatePath: string,
  vars: Record<string, unknown>
): { valid: boolean; errors: string[] } {
  const fileContent = fs.readFileSync(filePath, 'utf-8');
  const expectedMarkers = buildExpectedMarkers(templatePath, vars);
  const errors: string[] = [];
  let searchFrom = 0;

  for (const marker of expectedMarkers) {
    const idx = fileContent.indexOf(marker, searchFrom);
    if (idx === -1) {
      errors.push(`Missing marker: "${marker}"`);
    } else {
      searchFrom = idx + marker.length;
    }
  }

  return { valid: errors.length === 0, errors };
}
```

## Testing

### Unit Test: Freshly Generated File

```typescript
// my-generator/generator.spec.ts

import { validateConformance } from '../../tools/validators/template-conformance';
import * as path from 'path';

const TEMPLATE = path.join(__dirname, 'files/src/__name__.service.ts__template__');
const VARS = { name: 'user', className: 'User' };

it('generated file conforms to template', () => {
  const { valid, errors } = validateConformance(
    'libs/user/src/user.service.ts',
    TEMPLATE,
    VARS
  );

  expect(errors).toEqual([]);
  expect(valid).toBe(true);
});
```

### Unit Test: After Developer Implementation

Commit a fixture file representing a realistic implemented version of the generated file, then assert it still conforms:

```typescript
it('implemented file still conforms to template', () => {
  const { valid, errors } = validateConformance(
    path.join(__dirname, '__fixtures__/user.service.implemented.ts'),
    TEMPLATE,
    VARS
  );

  expect(errors).toEqual([]);
  expect(valid).toBe(true);
});
```

## CI Check (Optional)

To run conformance validation across all generated files in the workspace:

```typescript
// tools/scripts/check-template-conformance.ts

import * as glob from 'glob';
import { validateConformance } from '../validators/template-conformance';
import { resolveTemplateAndVars } from '../validators/template-registry';

const files = glob.sync('libs/**/*.service.ts');
let failed = false;

for (const file of files) {
  const resolved = resolveTemplateAndVars(file);
  if (!resolved) continue;

  const { valid, errors } = validateConformance(file, resolved.template, resolved.vars);

  if (!valid) {
    console.error(`\n❌ ${file} does not conform to its template:`);
    errors.forEach((e) => console.error('  ', e));
    failed = true;
  }
}

if (failed) process.exit(1);
else console.log('✅ All files conform to their templates.');
```

`resolveTemplateAndVars` is a lookup you implement to map a file path back to its generator template and the variables it was generated with (e.g. from a manifest written at generation time, or by parsing the filename).

## Tradeoffs

| Consideration | Detail |
| ------------- | ------ |
| **Collision risk** | Emoji + noun phrase markers are unlikely to collide with developer-written comments. Avoid bare single-emoji markers. |
| **Marker mutation** | If a developer edits `// 🔒 Private` to `// 🔒 Private fields`, the validator breaks. Consider matching on emoji prefix only (`// 🔒`) for looser but more resilient checks. |
| **Structural boilerplate** | Only section markers are validated — not surrounding boilerplate like class declarations or decorators. If that matters, keep critical lines outside any free zone and validate them as additional fixed markers. |
| **Variable substitution** | The template is rendered with known vars before extracting markers, so markers containing EJS expressions (e.g. `// 🔒 <%= className %> internals`) are handled correctly. |
| **No deep content validation** | This approach does not verify that any particular code exists within a section, only that the section headers are present in order. |
