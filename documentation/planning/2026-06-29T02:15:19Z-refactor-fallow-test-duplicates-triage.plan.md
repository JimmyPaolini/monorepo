# Introduction

![Status: Completed](https://img.shields.io/badge/status-Completed-brightgreen)

This plan defines a targeted refactor of duplicate code in test files only, based on a fresh fallow baseline. It explicitly scores duplicate families for refactor value, preserves intentional duplication (especially conformance generator parity tests), and assigns a concrete refactoring strategy (helper extraction or test.each) for each selected family.

## 1. Requirements & Constraints

- **REQ-001**: Use a fresh baseline generated by `pnpm exec nx run monorepo:fallow-duplicates --configuration=markdown` before selecting families.
- **REQ-002**: Scope includes all projects represented in duplicate test families: `applications/caelundas`, `applications/lexico-ingestion`, `tools/synchronization`, and `tools/conformance`.
- **REQ-003**: Refactor only high-value duplicate families; do not force deduplication of all families.
- **REQ-004**: For each selected family, decide and document whether `helper extraction` or `test.each` is the primary strategy.
- **REQ-005**: Keep test behavior and assertions semantically equivalent after deduplication.
- **SEC-001**: Do not introduce test helpers that bypass validation of file-system writes, CLI side effects, or user-facing command output.
- **CON-001**: Restrict scope to test files and test support utilities; do not include production-source deduplication families (for example Family 18 and Family 57).
- **CON-002**: Treat Family 50-56 conformance generator/validator parity tests as preserve-by-default; allow edits only when they are readability-only and do not reduce per-generator contract explicitness, assertion specificity, or behavioral coverage.
- **CON-003**: Keep project boundaries intact; do not create cross-project testing packages.
- **GUD-001**: Follow monorepo Nx-first workflow and run validation targets through Nx.
- **GUD-002**: Prefer readable tests over maximal DRY; stop deduplicating when abstraction obscures scenario intent.
- **PAT-001**: Use project-local `testing/` helpers for repeated setup/teardown and fixture construction.
- **PAT-002**: Use `test.each` for repeated assertion matrices where only input/output tuples differ.
- **PAT-003**: Maintain unit/integration/end-to-end naming conventions and existing test target structure.
- **DEC-001**: Baseline used for this plan reports 79 clone groups, 64 families, and 8192 duplicated lines (7.8%) across 168 files.
- **DEC-002**: Post-refactor verification reports 70 clone groups, 61 families, and 7858 duplicated lines (7.5%) across 166 files.

#### Triage Rubric (Objective Scoring)

- **RUBRIC-001 (line count score)**: Assign `lineScore` from duplicated lines per family: `>= 60 => 3`, `30-59 => 2`, `15-29 => 1`, `< 15 => 0`.
- **RUBRIC-002 (instance count score)**: Assign `instanceScore` from clone groups/instances per family: `>= 6 => 3`, `4-5 => 2`, `2-3 => 1`, `<= 1 => 0`.
- **RUBRIC-003 (setup-boilerplate repeatability score)**: Assign `setupScore` by repeated setup ratio across family snippets: `>= 0.50 => 3`, `0.30-0.49 => 2`, `0.15-0.29 => 1`, `< 0.15 => 0`.
- **RUBRIC-004 (semantic coupling penalty)**: Assign `couplingPenalty`: `low => 0`, `medium => -1`, `high => -2`.
- **RUBRIC-005 (readability-risk penalty)**: Assign `readabilityPenalty`: `low => 0`, `medium => -1`, `high => -2`.
- **RUBRIC-006 (priority formula)**: Compute `priorityScore = lineScore + instanceScore + setupScore + couplingPenalty + readabilityPenalty`.
- **RUBRIC-007 (selection thresholds)**: `priorityScore >= 6 => select for refactor`; `4-5 => defer unless local owner requests`; `<= 3 => preserve/defer`.
- **RUBRIC-008 (strategy selection gate)**: For selected families, choose `helper extraction` when `setupScore >= 2`; choose `test.each` when setup differs only by input/output tuples and `setupScore <= 1`; choose `mixed` when both conditions appear in the same family.
- **RUBRIC-009 (preserve gate)**: Preserve by default when `semantic coupling = high` or `readability risk = high`, unless there is a documented bug-prevention reason to refactor.

## 3. Alternatives

- **ALT-001**: Refactor all 64 families in one pass. Rejected because risk and review complexity are too high for behavior-sensitive tests.
- **ALT-002**: Exclude synchronization to keep scope app-only. Rejected because cross-project command test duplication is a top-value target.
- **ALT-003**: Force `test.each` for every selected family. Rejected because many duplicates are setup/fixture heavy and better solved with helper extraction.
- **ALT-004**: Extract cross-project shared test package for command harnesses. Rejected because it introduces unnecessary coupling; project-local helpers are preferred.

## 7. Risks & Assumptions

- **RISK-001**: Over-abstraction of tests can reduce readability and hide domain semantics.
- **RISK-002**: Cross-project command-test similarities may diverge, causing brittle shared helper APIs.
- **RISK-003**: Fallow cache reuse can hide recent duplicate changes if baseline is not refreshed intentionally.
- **RISK-004**: Conformance test deduplication could weaken template-contract coverage if not constrained.
- **ASSUMPTION-001**: Duplicate families selected here represent the highest-value opportunities under current baseline and monorepo constraints.
- **ASSUMPTION-002**: Existing testing helper modules are acceptable extension points and do not require architecture changes.
- **ASSUMPTION-003**: Team preference is to preserve intentional generator-parity duplication in conformance tests.
