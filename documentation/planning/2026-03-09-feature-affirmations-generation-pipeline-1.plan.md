# Introduction

![Status: Completed](https://img.shields.io/badge/status-Completed-brightgreen)

Add a unified LangGraph pipeline to the affirmations notebook (`affirmations.ipynb`) that performs end-to-end generation: web research, reference document creation, and grammar-structured affirmation generation — all in a single graph flow. The pipeline produces 3 affirmations per (tarot card × grammar) combination, yielding 3,744 total affirmations (78 cards × 16 grammars × 3 each).

The unified 6-node graph is: **search → analyze_sources → write_reference → analyze_card → generate → validate**. The first three nodes subsume the existing ingestion pipeline (cells 3–5) so that all logic lives in one self-contained flow. The reference document is written to `output/tarot-card/` as a side effect, and affirmation output is written in both JSON and Markdown to `output/affirmations/`.

## 1. Requirements & Constraints

### Functional Requirements

- **REQ-001**: Generate 3 affirmations per (Subject × Grammar) combination for all 78 TAROT_CARDS × 16 GRAMMARS
- **REQ-002**: Each affirmation must conform to the grammatical form specified by the target `Grammar` (mood, voice, tense, aspect, person, number, polarity, deixis, form)
- **REQ-003**: Each affirmation must be thematically connected to the tarot card subject's meaning, symbolism, and correspondences
- **REQ-004**: Output both JSON (`Affirmation` Pydantic model → `model_dump_json`) and Markdown formats per card
- **REQ-005**: The pipeline must perform web research (SearxNG + Trafilatura), generate a reference document, and write it to `output/tarot-card/` before generating affirmations — all within a single graph invocation
- **REQ-006**: The LangGraph flow has exactly 6 nodes: search → analyze_sources → write_reference → analyze_card → generate → validate
- **REQ-007**: Validation node rejects affirmations that do not match structural constraints of the target Grammar and retries generation (max 2 retries before skipping)

### Security Requirements

- **SEC-001**: No user-supplied input is passed directly to shell commands or filesystem paths — all slugs are pre-defined constants
- **SEC-002**: File writes use only pre-validated `Subject.slug` and `Grammar.slug` values for path construction

### Constraints

- **CON-001**: LLM is Ollama with Qwen 3.5 (0.8b/4b/9b) — supports native tool calling and structured output via `with_structured_output`
- **CON-002**: CPU-only inference — expect 30–60s per LLM call with qwen3.5:4b. Total batch includes 6 LLM calls per card (analyze_sources + write_reference + analyze_card + 16× generate + 16× validate) plus web search I/O
- **CON-003**: Context window: Qwen 3.5 supports 32K tokens. Research input and prompts should still be concise to reduce inference time on CPU
- **CON-004**: Python project uses pyright strict mode — all functions need explicit return types, no `Any` types
- **CON-005**: Ruff linting with rule set `I`, `E`/`F`, `UP`, `B`, `SIM`, `TCH` — imports auto-sorted
- **CON-006**: Nx `commands` array (not `command` string) in project.json to override TypeScript-centric targetDefaults

### Guidelines

- **GUD-001**: Follow the existing StateGraph + TypedDict pattern established in notebook cell 5 (CoT ingestion graph)
- **GUD-002**: New prompt templates go in `src/prompts.py` following the existing `ChatPromptTemplate.from_messages` pattern
- **GUD-003**: New Pydantic models go in a new `src/models.py` module
- **GUD-004**: Reuse `strip_markdown_fences()` and `write_document()` from existing notebook cells (extract to `src/output.py` if appropriate)
- **GUD-005**: Use `inflect` for plural forms consistent with existing usage
- **GUD-006**: Notebook cells should be self-contained and idempotent — re-running a cell should not corrupt state

### Patterns

- **PAT-001**: LangGraph StateGraph with TypedDict state — nodes return `dict[str, ...]` partial updates
- **PAT-002**: `llm.with_structured_output(PydanticModel)` for affirmation generation to enforce JSON schema compliance
- **PAT-003**: `operator.add` reducer on list state keys for accumulating affirmations across grammar iterations
- **PAT-004**: LangGraph `Send` API for map-reduce fan-out across grammars (generate affirmations for each grammar in parallel within a single card's pipeline)
- **PAT-005**: Prompt chaining — each node's output feeds the next node's input via state

## 3. Alternatives

- **ALT-001**: **Separate ingestion and generation pipelines** — Keep the existing 2-node ingestion graph (cells 3–5) and add a separate 4-node affirmation graph that reads pre-existing reference docs. Rejected because it requires running two pipelines manually and assumes reference docs already exist. A unified graph is simpler to operate and guarantees fresh reference documents.
- **ALT-002**: **4-node pipeline reading existing reference docs** — Skip web search and use pre-existing `output/tarot-card/` reference documents as input. Rejected because it creates a hidden dependency on a prior pipeline run, and stale reference documents would silently degrade affirmation quality.
- **ALT-003**: **Pipeline without validation** — Simpler, faster, but the user explicitly requested validation. Small models are prone to grammatical inconsistencies (e.g., generating a past-tense affirmation when future-tense was requested). Validation catches this.
- **ALT-004**: **LangGraph Send API for parallel grammar generation** — Fan-out to all 16 grammars simultaneously within a card's pipeline. Theoretically faster, but Ollama with CPU inference runs sequentially anyway (one request at a time). Sequential generation within the generate node is simpler and equally fast given hardware constraints.
- **ALT-005**: **ReAct agent with tools** — Using `create_react_agent` with research/generation tools. Rejected because the flow is deterministic (no dynamic tool selection needed). A fixed StateGraph is more predictable and debuggable for batch content generation.
- **ALT-006**: **Separate notebook for affirmation generation** — Creating a new `affirmation-generation.ipynb`. Kept in single notebook for now since both pipelines share setup cells (LLM, search) and the unified graph is the primary flow. Can be split later if the notebook grows too large.

## 7. Risks & Assumptions

### Risks

- **RISK-001**: **Structured output edge cases** — Even with Qwen 3.5's native tool calling, the smallest model (0.8b) may occasionally produce malformed JSON. **Mitigation**: `with_structured_output` includes built-in retry via LangChain's error handling. If persistent, use the 4b model.
- **RISK-002**: **Grammar conformance accuracy** — The LLM may generate affirmations that don't precisely match the target grammatical form (e.g., wrong tense/mood). **Mitigation**: Validation node (TASK-011) catches mismatches. Retry up to 2 times. On final failure, keep the affirmation but flag it.
- **RISK-003**: **Batch processing time** — 78 cards × (1 web search + 5 LLM calls + 16× generate + 16× validate) = ~2,500+ LLM invocations plus network I/O. At 30–60s per LLM call on CPU, total time is substantial. **Mitigation**: Resume logic (TASK-015) skips cards whose affirmation output already exists. Run in background or overnight sessions.
- **RISK-004**: **Web search availability** — SearxNG must be running for every pipeline invocation. If SearxNG is down or returns empty results, the entire pipeline for that card fails. **Mitigation**: The `search` node raises a clear error if no content is extracted. SearxNG can be restarted via `nx run affirmations:searxng`.
- **RISK-005**: **Reference document quality** — Since reference documents are generated fresh within the same pipeline, quality depends on web search result quality and LLM summarization. **Mitigation**: The two-step CoT (analyze_sources → write_reference) is already proven to produce better results than one-shot ingestion.
- **RISK-006**: **Inference time on CPU** — Qwen 3.5 9b is significantly slower on CPU than 0.8b/4b. **Mitigation**: Use 0.8b or 4b for batch processing; reserve 9b for single-card quality runs. The analyze_card node (TASK-009) still compresses the reference document into a ~300 word brief to minimize token count per generation call.

- **ASSUMPTION-001**: The Ollama service is running and accessible at `http://host.docker.internal:11434` (or via Docker Compose service name).
- **ASSUMPTION-002**: The SearxNG service is running and accessible at `http://searxng:8080` for web search.

- **ASSUMPTION-004**: 3 affirmations per grammar is sufficient variety. This can be adjusted by changing a constant.
